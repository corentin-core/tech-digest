📖 En détail

1. sécu $1 500 d’expérience : quels LLM hackent une vraie app ?

L’app est de l’ordre du réaliste : API FastAPI bien durcie, mais le bundle Android contient un google-services.json qui expose un projet Firebase dont les règles Firestore sont ouvertes. Le pattern (« API blindée + Firebase grand ouvert ») est ce que l’auteur croise en audit. Chaque modèle a 10 runs, $10 max et 2 h pour récupérer un flag dans les private reviews d’un user.

modèle solve $/solve tokens/run médians
gpt-5.5 7/10 9,46 $ 260 k
deepseek-v4-pro 3/10 0,62 $ 194 k
claude-sonnet-4.6 2/10 45,75 $ 390 k
claude-opus-4.8 2/10 16,15 $ 113 k
gemini-3.1-pro / 3.5-flash 0/10 — refus

Quatre observations qui transfèrent : - GPT-5.5 attaque direct le bon vecteur (déballer l’APK → Firebase). La plupart des autres tournent sur l’API REST et n’en sortent pas → l’écart est une question d’intuition d’archi, pas de capacité brute. - Claude Opus 4.8 se refuse en cours de tâche (« late refusals ») près de la solution — pénalise un agent qui réfléchit en profondeur. - Modèles chinois (DeepSeek, Kimi, MiniMax) sans inhibition sur les actions destructives (« ça toucherait la vraie DB ») — les autres marquent un blip moral. - Le $/solve écrase la solve rate : DeepSeek V4 Pro à 0,62 $/solve vs Sonnet 4.6 à 45,75 $. Même quand Claude réussit, c’est ~75× plus cher.

⚠️ L’auteur le dit lui-même : pas un eval scientifique (10 runs, harness différent pour Claude — Claude Code -p —, températures non uniformes selon ce qu’accepte le modèle). À lire comme retour terrain coloré, pas comme leaderboard.

Pour toi : la leçon vaut au-delà des LLM. Quand un agent (ou un audit) te dit « API blindée », regarde à côté de l’API — services tiers (Firebase, Supabase, S3 mal scopé), bundles client, secrets dans le binaire. Le périmètre audité n’est jamais celui que tu crois, et l’erreur typique est de croire à un mur quand il y a une porte de service ouverte deux mètres plus loin.

2. systèmes micropython-wasm : sandbox Python via WebAssembly

Le besoin : exécuter du Python fourni par l’utilisateur (plugin, scripting, code généré par agent) sans monter un conteneur ni démarrer un sous-processus. La recette : - MicroPython compilé en WebAssembly (un blob WASM de ~362 Ko, 78 lignes de C custom pour exposer les fonctions hôte). - Exécuté par le moteur wasmtime côté Python. - Limite mémoire directe (supportée par wasmtime). - Limite CPU via le mécanisme fuel — un compteur d’instructions WASM décrémenté à chaque opération. Pratique, mais l’unité « fuel » est dure à mapper sur du temps réel. - État persistant entre session.run() via threads + queues : l’interpréteur reste chaud d’un appel à l’autre. - Pas de système de fichiers ni de réseau par défaut — il faut exposer explicitement.

Simon Willison précise : « je ne recommanderais ça à personne qui n’accepte pas un risque significatif » — implémentation alpha, en partie vibe-coded, qui repose sur la maturité de WebAssembly côté navigateur, pas sur un audit sécu. GPT-5.5 a essayé des évasions sans succès, mais ce n’est pas une validation.

⚠️ Le titre « sandbox Python » sonne mûr ; lire « prototype prometteur avec primitives natives intéressantes ».

Pour toi : point d’archi à avoir en tête quand quelqu’un te demandera « comment je laisse un agent Claude exécuter du Python tiers sans Docker ? ». Et un fil concret côté WASM si tu veux ouvrir cette porte au-delà de ruxe — wasmtime + WASI sont la couche en dessous de plein de chaînes d’agents.
