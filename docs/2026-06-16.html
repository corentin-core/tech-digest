📖 En détail

1. bdd TimescaleDB Hypercore : moteur hybride lignes + colonnes

Hypercore résout un dilemme classique des bases de séries temporelles : on veut écrire vite (insertions ligne par ligne, format orienté lignes) et lire/agréger vite sur l’historique (format orienté colonnes, bien plus compact). La solution est de ne pas choisir : les données fraîches atterrissent dans des chunks (tranches de table) classiques en lignes, et dès qu’une tranche est « refroidie » (plus modifiée), elle est convertie automatiquement en colonnes compressées. On regroupe alors ~1 000 lignes en une seule ligne « compressée » où chaque colonne devient un tableau — c’est ce regroupement en lots qui permet la compression.

Le cœur, c’est que la compression est choisie par type de colonne : encodage delta et delta-de-delta (on stocke l’écart entre valeurs successives plutôt que la valeur absolue) plus simple-8b pour les horodatages et entiers ; compression XOR inspirée de Gorilla pour les flottants de capteurs (qui bougent peu d’une mesure à l’autre) ; dictionnaire + encodage par plages (RLE) pour le texte répétitif. Exemple parlant : une colonne de température stocke 72.5, +0.2, -0.3, +0.2… au lieu des valeurs entières — beaucoup moins d’octets car les écarts sont minuscules.

Deux paramètres pilotent tout : segmentby (regrouper les lignes partageant une même valeur, p. ex. l’identifiant de machine) et orderby (trier dans le lot, p. ex. par temps décroissant, pour maximiser l’efficacité du delta). Le piège à connaître : si la colonne segmentby a trop de valeurs uniques avec peu de lignes chacune, les lots se remplissent mal et la compression s’effondre. La doc recommande au moins 100 lignes par segment, idéalement 100 à 10 000 valeurs uniques. Sur le seul cas réel montré (données capteurs MQTT, ~180 identifiants), la requête passe de 10,2 ms à 0,36 ms (≈ 28× plus vite à l’exécution) et 308 Mo tombent à 7,2 Mo (≈ 43× de compression).

⚠️ Le titre « jusqu’à 98 % » est survendu : ce chiffre n’apparaît dans aucun banc d’essai de l’article. L’auteur lui-même prévient que ses 43× (97,7 %) viennent d’un jeu de données « à redondance exceptionnelle », et qu’une attente réaliste sur des séries temporelles typiques est plutôt 8 à 20× (92–95 %). Honnête sur ses propres limites, ce qui est rare. En pratique : pertinent dès qu’on stocke des séries temporelles à fort volume (métriques, télémétrie, capteurs) sous PostgreSQL et qu’on veut éviter de sortir une base colonne dédiée ; le réglage de segmentby/orderby est exactement le genre de détail de tuning perf qui fait 10× d’écart.

2. sécu Une porte dérobée cachée dans une offre d’emploi LinkedIn

L’attaque est une belle leçon d’ingénierie sociale couplée à un abus de mécanisme. Un faux recruteur — usurpant l’identité LinkedIn réelle d’un journaliste — envoie un dépôt GitHub en demandant « de jeter un œil au souci de modules Node dépréciés ». L’appât n’est pas le code lui-même mais l’instruction implicite : lancer npm install. Or npm exécute automatiquement le script prepare à l’installation, qui ici appelait node app/index.js. Ce fichier chargeait app/test/index.js, où se cachait — sur une seule ligne minifiée, noyée au milieu de code de test commenté — la charge utile : elle contactait un serveur (rest-icon-handler.store) et exécutait tout ce que celui-ci renvoyait. Autrement dit, exécution de code distant arbitraire dès l’installation, sans jamais lancer l’application.

Le point non-évident, c’est le vecteur : pas de faille technique exotique, juste le cycle de vie standard de npm (prepare/postinstall) détourné. Tout développeur qui « teste rapidement » un dépôt inconnu se fait avoir avant même d’avoir lu une ligne. La parade décrite est élégante : l’auteur n’a pas installé en local, il a cloné le dépôt sur un serveur jetable (VPS) puis lâché dessus un agent IA en lecture seule (outils restreints à read,grep,find,ls). L’agent a tiqué quasi immédiatement sur le comportement suspect. Le titre ne survend rien : l’attaque est réelle et le mécanisme exact. En pratique : réflexe utile pour quiconque audite des dépendances ou des dépôts tiers — désactiver les scripts d’installation (npm install --ignore-scripts), inspecter dans un bac à sable jetable, et garder en tête que postinstall/prepare sont une surface d’exécution de code, pas juste de la plomberie.

3. IA Petit guide de l’ingénierie de l’inférence des LLM

L’idée maîtresse, et c’est elle qui vaut la lecture, est que servir un grand modèle de langage se découpe en deux phases aux goulets d’étranglement opposés. La phase de prefill (préremplissage) ingère tout le prompt d’un coup à travers toutes les couches du modèle : elle est limitée par le calcul — ce sont les unités arithmétiques du GPU qui saturent. La phase de decode (génération) produit les jetons un par un, une passe complète par jeton : elle est limitée par la bande passante mémoire — le GPU passe son temps à relire les poids du modèle depuis la mémoire pendant que ses unités de calcul tournent à vide. Entre les deux, le cache KV (les valeurs d’attention intermédiaires) est calculé une fois au prefill puis réutilisé à chaque jeton généré.

Cette grille de lecture range d’un coup toutes les optimisations qu’on entend citer pêle-mêle. Le batching (entrelacer plusieurs requêtes) sacrifie la latence individuelle pour saturer le calcul. Le cache de préfixe réutilise le cache KV quand des prompts partagent un même début. La quantification (compresser les poids en 4–8 bits) attaque directement le mur mémoire du decode — annoncée à « 30 à 50 % » de gain. Le décodage spéculatif fait deviner plusieurs jetons par un petit modèle, vérifiés en une passe par le gros. Une fois qu’on sait quel goulet on attaque, on cesse de traiter toutes ces techniques comme interchangeables.

⚠️ À calibrer : c’est un cadre conceptuel, pas un guide d’implémentation. Le « ~80 % de réduction de coût à grande échelle » est cité sans démonstration, et des termes attendus (paged attention, continuous batching) ne sont pas traités sous ces noms. Utile pour l’architecte qui veut la bonne carte mentale, pas pour l’ingénieur qui code le moteur. En pratique : le bon modèle mental pour raisonner sur le coût/latence d’un service d’inférence (y compris en local) — savoir si on est limité calcul ou mémoire avant de choisir une optimisation, exactement le même réflexe que pour tout code perf-critique en C++/Rust.
