📖 En détail

1. perf SQLite : trier avant d’insérer

Le constat est simple et mesuré : sur l’insertion de 10 millions de lignes (clés primaires aléatoires, blobs de 20 octets), passer par un tri préalable fait tomber le temps de 11 103 ms à 3 835 ms, soit un facteur ~2,9× — d’environ 900 000 insertions/s à ~2,6 millions/s. Et ce, malgré le coût du tri lui-même, qui est largement remboursé.

Le mécanisme tient à la structure de stockage de SQLite, un B+ tree (l’arbre équilibré qui range les lignes par ordre de clé sur des « pages » disque). Insérer dans l’ordre des clés, c’est toujours écrire « à la fin » de la page courante : séquentiel, donc rapide. Insérer des clés aléatoires, c’est au contraire viser sans cesse des pages déjà pleines un peu partout dans l’arbre, ce qui force des éclatements de page (page splits : couper une page pleine en deux pour faire de la place) et des rééquilibrages de l’arbre. Ces opérations multiplient les écritures réelles par rapport aux données logiques — le phénomène d’amplification d’écriture. En triant en amont, on supprime presque tout ce remue-ménage.

La nuance honnête : le banc d’essai reste agrégé. On a le temps total, pas la mesure directe du nombre d’éclatements de page, des I/O disque ou du comportement de cache — et un seul format de données (blobs de 20 octets, clés aléatoires). Le gain est donc démontré dans ce cas (insertion par lots de données non ordonnées) mais pas disséqué. À noter aussi : si vos données arrivent déjà à peu près triées, le bénéfice fond.

En pratique : utile dès qu’on alimente une base embarquée par gros lots — ingestion de séries temporelles, import de fichiers, ETL local. Le réflexe « trier par clé primaire avant le batch insert » est quasi gratuit en code et se transpose à d’autres moteurs à B-tree (le même raisonnement vaut pour les index en général).

2. systèmes La variable Git false_but_the_compiler_does_not_know_it_

Git embarque, dans un fichier source à part, un int false_but_the_compiler_does_not_know_it_ = 0; et une macro #define NOT_CONSTANT(expr) ((expr) || false_but_the_compiler_does_not_know_it_). Le but : éteindre un faux positif de l’avertissement Clang -Wunreachable-code (« ce code ne sera jamais exécuté »).

Le problème de fond est typique du C multi-configuration. Selon les options de compilation, une fonction peut être remplacée par une macro constante : avec NO_SYMLINK_HEAD défini, create_ref_symlink(...) devient littéralement (-1). Du coup un if (!(-1)) continue; paraît mort pour cette configuration, et le compilateur râle — alors que dans les autres builds, ce code est bien atteignable. On ne veut ni supprimer le code, ni désactiver l’avertissement partout.

L’astuce exploite finement le modèle de compilation du C : la variable est non-const, à liaison externe, définie dans une autre unité de compilation. Quand le compilateur traite le fichier qui utilise NOT_CONSTANT, il voit la variable mais ne peut pas prouver qu’elle vaut toujours 0 (elle pourrait être modifiée ailleurs). Il est donc obligé de garder la branche : plus d’avertissement « code mort ». L’élégance : à l’édition de liens avec LTO (link-time optimization, l’optimisation qui voit tout le programme d’un coup), le compilateur constate enfin que la variable reste nulle et élimine quand même la branche du binaire final. On a donc le beurre (silence du compilateur) et l’argent du beurre (code mort supprimé au final).

En pratique : une alternative chirurgicale à volatile ou aux #ifdef pour neutraliser une analyse statique trop zélée, sans changer la sémantique. Bon à connaître pour quiconque maintient du C/C++ portable compilé sous plusieurs toolchains et configurations.

3. IA « Is Grep All You Need ? » — la plomberie prime sur la recherche

L’article s’appuie sur une étude (PwC) qui teste 116 questions du jeu LongMemEval — des questions de rappel sur de la mémoire conversationnelle — en comparant recherche lexicale (grep, mots-clés) et recherche vectorielle (embeddings « sémantiques »). Verdict apparent : grep gagne. Quelques chiffres, à livraison « inline » et harnais Chronos : Claude Opus 4.6 à 93,1 % (grep) contre 83,6 % (vecteurs) ; Gemini 3.1 Flash-Lite 86,2 % contre 62,9 % (23 points d’écart).

Mais le vrai enseignement est ailleurs, et il est plus profond. À modèle, corpus et méthode de recherche identiques, changer seulement le harnais (la façon dont l’agent reçoit et manipule les résultats) fait bouger Opus 4.6 de 93,1 % à 76,7 % — 16 points, juste sur la « plomberie ». Plus spectaculaire : GPT-5.4 tombe de 93,1 % (résultats livrés inline, dans le contexte) à 55,2 % (livrés via fichiers), à recherche rigoureusement identique. Autrement dit, comment on présente l’information au modèle pèse autant, voire plus, que par quelle méthode on l’a trouvée.

⚠️ Putaclic partiel : le titre laisse croire que « grep gagne toujours ». Les auteurs eux-mêmes nuancent — le banc d’essai porte sur du rappel d’extraits présents mot pour mot ; pour de la synthèse ou du raisonnement sur des formulations paraphrasées, le vectoriel peut reprendre l’avantage. Le résultat solide n’est pas « grep > vecteurs » mais « la conception du harnais rivalise avec le choix de la méthode de recherche ».

En pratique : directement parlant pour qui exploite des agents de code (Claude Code & co.) ou bâtit du RAG : avant d’investir dans des embeddings sophistiqués, soignez la livraison du contexte (inline vs fichiers, format, budget de contexte) — c’est souvent là que se cache le gain. Et méfiez-vous des bancs d’essai dont le périmètre ne colle pas à votre usage réel.
