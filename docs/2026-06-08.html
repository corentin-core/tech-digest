📖 En détail

1. Every byte matters

L’idée centrale : à complexité algorithmique égale (O(N)), la disposition des données en mémoire domine le temps réel parce que le CPU charge des cache lines de 64 octets. Si tu itères un Array of Structs où chaque struct fait 64B mais que tu ne lis qu’un booléen, tu gâches 63B de capacité de cache à chaque ligne. En Struct of Arrays (un tableau par champ), tu peux packer 64 booléens dans une seule ligne → lecture séquentielle dense.

Pour les accès aléatoires (hash maps, arbres), c’est la taille du working set qui décide du niveau de cache atteint : la table de l’article montre 512 structs de 64B qui tiennent en L1d (~3 ns), contre 128B qui débordent en L2 (~11 ns). Le takeaway actionnable : minimise la taille de tes structs sur les chemins chauds en accès aléatoire — doubler 64B→128B double le working set et te pousse vers un cache plus lent.

⚠️ Le « jusqu’à 30× » du titre est conditionnel à un struct de 1 KiB. Sur un cas réaliste 64B→128B, le gain mesuré est ~3,6× (3 → 11 ns). Et il n’y a aucun code ni méthodo de bench reproductible : c’est de la bonne intuition systèmes (diagramme de hiérarchie de cache à l’appui), pas une étude empirique rigoureuse.

2. How we contain Claude across products

Le principe directeur : des défenses qui se chevauchent (« quand la couche environnement n’est pas dispo, la couche modèle prend le relais »). Trois couches : - Environnement (primaire) : conteneurs gVisor (claude.ai), sandboxes OS type Seatbelt/bubblewrap (Claude Code), VMs complètes avec isolation vsock (Cowork). - Modèle (secondaire) : system prompts + classifiers qui attrapent les « overeager behaviors ». Le mode auto de Claude Code bloque ~83% de ces actions avant exécution — donc ~17% passent (les défenses probabilistes ont une limite par construction). - Outils/contenu : allowlists, proxy MITM, mounts (read-only, read-write, read-write-no-delete).

Détails qui transfèrent à n’importe quel déploiement d’agent : (1) la résolution des symlinks doit se faire avant la validation de chemin, sinon un symlink dans un dossier autorisé peut pointer dehors et s’échapper ; (2) un allowlist egress vers api.anthropic.com n’est pas un filtre de destination mais une capacité d’exfiltration — ils ont dû mettre un proxy MITM dans la VM qui ne laisse passer que le token de session de la VM et rejette les clés API volées ; (3) la fatigue d’approbation est réelle (taux d’acceptation de 93%) → ne jamais compter sur les dialogues seuls, doubler avec de l’enforcement OS.

Pas de putaclick : vraie profondeur d’ingénierie. 5 incidents documentés en post-mortem, et la leçon « la couche la plus faible est celle que tu as codée toi-même » (leurs proxys maison ont cédé, les primitives éprouvées — hyperviseurs, seccomp, gVisor — ont tenu).
