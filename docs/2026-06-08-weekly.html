📖 En détail

1. perf Every byte matters

À complexité égale, la façon dont les données sont rangées en mémoire domine le temps réel, parce que le processeur charge la mémoire par blocs de 64 octets (cache line, « ligne de cache »). Parcourir un tableau d’objets de 64 octets pour ne lire qu’un booléen gaspille 63 octets de cache à chaque ligne. En rangeant les données par champ (un tableau de tous les booléens, un autre de tous les noms…), chaque ligne de cache se remplit de données utiles. Pour les accès aléatoires, c’est la taille du working set (les données réellement touchées à un instant donné) qui décide du niveau de cache atteint : 512 objets de 64 o tiennent dans le cache L1 (~3 ns), 128 o débordent dans le L2 (~11 ns).

⚠️ Le « jusqu’à 30× » du titre suppose des objets de 1 Ko ; sur un cas réaliste (64→128 o) le gain tombe à ~3,6×, et il n’y a aucun banc d’essai reproductible — bonne intuition, pas une étude rigoureuse.

Pour toi : sur tes boucles chaudes en C++ (traitement de séries temporelles EMS), passer d’un tableau d’objets à des tableaux par champ peut couper les défauts de cache sur les parcours séquentiels — exactement le raisonnement de l’article.

2. perf Niveaux de microarchitecture amd64 en Go

Daniel Lemire mesure l’effet de recompiler pour un jeu d’instructions plus récent (variable GOAMD64 : v1 = base, v2 = +popcnt/SSE4.2, v3 = +AVX2, v4 = +AVX-512 — popcnt compte les bits à 1 ; AVX = instructions SIMD, c.-à-d. qui traitent plusieurs valeurs en une seule opération). Sur la bibliothèque Roaring Bitmaps (Xeon Gold) : comptage de bits −43 % de v1 à v2, une autre opération −38 % en v3, une troisième +22 %. v4 (AVX-512) n’apporte presque rien.

Pour toi : l’équivalent en Rust/C++ c’est compiler avec RUSTFLAGS=-C target-cpu=native (ou -march=native) — tu actives les instructions SIMD de ta machine sans toucher au code. À éviter pour un binaire distribué (il planterait sur un CPU plus ancien), pertinent pour un service déployé sur une machine que tu maîtrises.

3. distribué Pourquoi les files d’attente ne règlent pas la surcharge

Le réflexe classique : ajouter une file pour « absorber » la charge. Mais par la loi de Little (taille de la file = débit d’arrivée × temps de traitement), si les arrivées dépassent la capacité, la file croît vers l’infini. S’ensuit une spirale : files qui gonflent → expirations → ré-essais → encore plus d’arrivées → pression mémoire → pauses du ramasse-miettes → pire encore. La file ne fait que masquer la surcharge et retarder l’effondrement.

La vraie réponse : le load shedding (rejeter délibérément des requêtes dès que le système sature) — files bornées, rejet immédiat, expirations obligatoires — pour se dégrader proprement plutôt que crasher en saturation mémoire. Rendre la surcharge visible quand elle arrive.

Pour toi : pertinent pour tes flux temps réel EMS — quand un consommateur sature (écriture en base, dialogue onduleur), mieux vaut refuser proprement que laisser une file enfler jusqu’à la panne.

4. sécu Vol de jeton GitHub en « 1 clic » via un bug VSCode

Pas un simple piège web : une vraie chaîne d’attaque sur les webviews de VSCode (pages web embarquées dans l’éditeur). Un notebook piégé (sur github.dev) exécute du JavaScript qui simule des frappes clavier — VSCode les traite comme de vraies touches de l’utilisateur. Via des raccourcis, le script installe une extension locale (qui contourne les contrôles de confiance) ; l’extension lit alors le jeton d’authentification GitHub et liste tes dépôts privés.

⚠️ « 1 clic » survend : il faut s’être déjà connecté à github.dev au préalable.

Pour toi : tu vis dans VSCode + agents — la leçon : des frappes clavier simulées peuvent déclencher des actions « de confiance », donc prudence avec ce qu’on ouvre dans l’éditeur.
